## 做T（日内交易）操作框架

当用户要做T+0日内交易时，**必须提供精确价位**，不能只给方向性建议。

### 操作计划模板

**⚠️ 统一输出格式（2026-06-28 修正）**：每只ETF必须同时给出买入点和卖出点（配对），格式如下：

```
买入点: X.XXX → 卖出点: X.XXX（价差 X.XX%）
```

**禁止**以下格式：
- ❌ 范围格式："1.085~1.090"（模糊，无配对）
- ❌ 多档格式："买点1: XX元 / 买点2: XX元"（冗长）
- ❌ 只有买点没有卖点，或只有卖点没有买点

**示例**：
```
📌 建议: 做T降本
💰 买入点: 1.120 → 卖出点: 1.128（价差 0.71%）
⛔ 止损价: 1.087
🎯 大概率卖出价位: 1.125(卖一) / 1.125~1.128(买一上方)
```

### 做T降本核心逻辑
- **高卖低买**：卖出价必须 > 买入价，否则不做
- **分批操作**：每次卖1/3仓位，不卖全（防T飞）
- **价差要求**：买卖价差 ≥ 0.5%（覆盖手续费+滑点）
- **尾盘清仓**：除非强势涨停，否则14:50前平掉当天买入部分

### 做T演算公式
```
新均价 = (原克数×原均价 - 卖出克数×卖出价 + 买入克数×买入价) ÷ 总克数
每股利润 = 卖出价 - 买入价
总利润 = 每股利润 × 卖出克数
```

### ⚠️ 特殊场景处理

#### 1. LOF折溢价做T策略修正
```
溢价率 = 场内价格 / 净值 - 1

| 溢价率 | 做T策略 | 买入点 | 卖出点 |
|--------|---------|--------|--------|
| > 5%（高溢价） | ⚠️ 暂停加仓做T | 场内价格 | 等溢价收窄后卖出 |
| 0~5%（正常） | 正常做T | 买一/买二 | 卖一/卖二 |
| < -5%（高折价） | ✅ 折价是做T好时机 | 场内价格（更便宜） | 等价格回归净值 |

⚠️ 高溢价时继续加仓做T会加剧溢价，最终场内价格回归净值时产生额外亏损。
```

#### 2. 分红除息成本调整
```
调整后成本 = 原成本 - 每份分红金额

⚠️ 持仓期间若ETF/LOF分红除息，必须调减成本价。
例如：518880黄金ETF成本9.6332，若分红0.4元/份 → 调整后成本9.2332
分红后重新计算保本价和做T区间。
```

#### 3. 流动性枯竭
- 买一量 < 持仓量的1% → 暂停做T建议，标注"流动性不足，暂不建议操作"
- 买卖价差 > 0.5% → 降低单次做T仓位（从1/3降为1/4）

#### 4. 止损价统一公式（2026-06-28 修正）
```
止损价 = 成本价 × 0.95（5%无条件清仓线）
```
- 所有ETF统一使用此公式，**禁止**使用固定数值或"跌破XX整数关口"等模糊说法
- 对于深套品种（浮亏 > 20%），标注"止损价仅作参考，用户不认亏卖，建议做T降本"
- 示例：成本2.8397 → 止损价 = 2.8397 × 0.95 = **2.698**

## 分析模板

对任意A股ETF，按以下结构输出分析报告：

### 1. 基本信息
- 基金全称、代码、成立日期、管理人
- 标的指数、风险等级(R1-R5)
- 规模、综合费率(管理+托管)
- **交易机制：T+0还是T+1**（必须标注！）

### 2. 行情数据
- 最新净值/价格、日涨跌幅
- 近1周/1月/3月/6月/1年涨跌幅
- 今年以来、成立以来收益率

### 3. 成分股分析
- 前十大权重股及占比
- 行业集中度
- 个股近期表现分化

### 4. 资金面信号
- 净流入/流出方向
- 融资买入 vs 净赎回矛盾
- 份额变化趋势（连续净申购/净赎回）

### 5. SWOT分析
| 维度 | 内容 |
|------|------|
| 优势(S) | 费率、流动性、政策催化、业绩趋势 |
| 劣势(W) | 集中度、规模偏小、跟踪误差 |
| 机会(O) | 估值位置、政策利好、行业景气 |
| 威胁(T) | 短期过热、资金流出、外部风险 |

### 5.5 行业ETF/LOF溢价分析（2026-06-28 新增）

#### 行业ETF板块轮动信号
```
板块轮动判断：
- 近5日涨幅 > 3% → 板块过热，谨慎追高
- 近5日跌幅 > 3% → 板块超跌，关注反弹机会
- 板块相对大盘（沪深300）超额收益 > 2% → 强势板块
```

#### LOF溢价率计算（适用于161226白银LOF等）
```
溢价率 = (场内价格 - 净值) / 净值 × 100%

| 溢价率 | 策略 | 说明 |
|--------|------|------|
| > 5% | ⚠️ 暂停做T | 溢价过高，场内价格虚高，等待回归 |
| 0~5% | 正常做T | 溢价合理，可正常日内操作 |
| < -5% | ✅ 折价买入 | 场内价格低于净值，套利机会 |
```

#### 跨市场ETF对比（同主题不同市场）
```
对比维度：
- 跟踪标的（是否同一指数）
- 交易机制（T+0 vs T+1）
- 溢价率（场内 vs 净值）
- 流动性（日均成交额）
- 费率（管理+托管）

示例：黄金ETF（518880 T+0）vs 白银LOF（161226 T+1）
→ 黄金T+0可日内做T，白银T+1只能隔日操作
→ 白银溢价率需额外关注，避免高溢价时加仓
```

## ⚠️ 输出约束：保本价强制检查（每条必做）

**在输出任何卖出建议前，必须先计算保本价并验证卖点 ≥ 保本价。**

```
保本价 = 成本价 × 1.0015   （成本 + 0.15%手续费）
```

### 检查规则

| 情况 | 卖点要求 | 输出方式 |
|------|---------|---------|
| 现价 ≥ 保本价 | 卖点1/2/3 可以正常给出（卖一上方0.001~0.005） | 正常输出三档卖点 |
| 现价 < 保本价 | **不允许给出低于保本价的卖出建议** | 改为："当前价低于保本价，无法保本卖。建议做T降本，目标价至少 {保本价:.4f}" |
| 浮亏 > 10% | 日内难以保本 | 增加提示："当前浮亏{X}%，日内难以保本，建议分批做T自救而非直接卖出" |

### 做T买入点验证
- 买入点必须 ≤ 现价 × 0.99（在当前价格附近能买到）
- 如果买入点需要下跌 > 3% 才能触发，标注"买入点较激进"
- T+0做T必须验证：卖出点 > 买入点（否则空间为负，不做）

### 配对验证
- 每只ETF的买入点和卖出点必须配对出现（买→卖，不能只有买或只有卖）
- 买入点 → 卖出点之间的价差 ≥ 0.5%（覆盖手续费+滑点）

## 用户偏好

- 用户关注具体数字，不要模糊表述
- 用户会提供成本价，需要计算浮盈/浮亏并给出操作建议
- **用户偏好极度简洁**：不要重复用户已知道的信息，不要解释为什么要买/卖，直接给方案（代码+价位+仓位）。一句话能说清的不用两句。
- 用户对"追高"持谨慎态度，强调风险信号
- 用户同时关注A股和港股ETF
- 用户要求操作建议必须包含精确价位（买入点、卖出点、止损点），不接受纯方向性建议。"你不给点位，我怎么卖？"——没有点位=没用
- 用户熟悉做T（日内高卖低买降成本）策略，需要提供具体的做T方案（分批比例、触发价位、目标利润）
- 用户对LOF（如161226白银LOF）也做T，需了解LOF溢价风险
- 用户不接受"现在涨了不能追"的模糊说法，必须给出具体点位和条件
- **用户会当场纠正价格错误**，说明用户有实时数据渠道，我如果给的现价≠用户看到的现价，说明我用的是过时数据，必须重新获取
- **用户风险偏好**：不接受债券/货币/国债类低波动ETF。可以接受短期波动（7天持有），但**不能接受单向持续下跌**（如161226白银LOF的溢价回归式跌）。要求：有涨有跌、T+0、不归零、7天内有反弹机会。黄金ETF是最符合的品种。
- **不认亏卖**：用户明确说"我不喜欢亏损卖，我可以用时间换空间"。即使浮亏很深（如-37%），也不建议清仓止损，而是通过做T降本、补仓摊薄来"用时间换空间"。只有当确认趋势彻底恶化（如归零风险）时才建议清仓。
- **每只ETF必须给出配对买卖点位**：不能只给买点不给卖点，也不能只给卖点不给买点。每只基金必须同时给出：买入点 → 卖出点（配对），以及对应的触发条件和仓位比例。
- **买卖点位必须是"大概率能成交"的价格**：基于买一/卖一价、盘口深度、历史支撑压力位给出实际可成交的价格，不要给理想化的数字。
- **卖出点位不能低于成本价**：用户投诉"这么低就可以卖了吗？白玩？"。所有建议卖点必须 ≥ 成本 × 1.0015（保本价），否则改为做T降本建议。

### 关键指标计算

```python
# 买卖盘总量
buy_total_vol = buy1_vol + buy2_vol + buy3_vol + buy4_vol + buy5_vol
sell_total_vol = sell1_vol + sell2_vol + sell3_vol + sell4_vol + sell5_vol

# 买卖比（>1.2买盘占优，<0.8卖盘占优）
buy_sell_ratio = buy_total_vol / sell_total_vol

# 大单定义（≥500手）
# 大单买入总量 vs 大单卖出总量 → 判断主力资金方向

# 买卖价差（反映流动性）
spread = sell1_price - buy1_price  # 通常0.001-0.003
```

### 盘口信号判读

| 信号 | 含义 | 操作建议 |
|------|------|---------|
| 买一量 >> 卖一量（3倍以上） | 场内接盘积极 | 上涨趋势强，可持有或挂高单卖出 |
| 卖一量 >> 买一量 | 抛压大 | 考虑降价或市价出 |
| 卖一~卖五大单密集 | 上方抛压重 | 提高卖单价格预期，不急于挂低价 |
| 买一挂了巨量（万手级） | 主力托底 | 现价下方有支撑，回调空间有限 |
| 买卖比 0.5-0.8 | 偏空 | 谨慎持有，设好止损 |
| 买卖比 1.5+ | 偏多 | 可加仓或挂高单 |

### 挂单成交概率评估

当用户挂了一个卖出价X时：
1. 若现价 ≥ X → 大概率已成交
2. 若卖一价 ≤ X → 成交概率高（有人愿意以更低价格接盘）
3. 若买一价距X只差0.001-0.005 → 中等概率，等买盘消化后上攻
4. 若买一挂了巨量（万手级）且X高于买一0.01以上 → 概率偏低，需等买盘被吃掉

## 关键风险信号识别

### 过热信号 ⚠️
- 近1周涨幅 > 5%
- 连续20日净赎回 + 价格上涨（价升量减）
- 融资净买入与净赎回矛盾
- 近1月/1季涨幅 > 15%

### 低估信号 ✅
- 近1年跌幅 > 5%
- 估值处于历史低位区间
- 资金持续流出但净值抗跌

### 获取实时价格（重要方法论）

**优先级**：
1. `web_search` 搜索 "{代码} ETF 今日行情 最低最高" — 最快最稳定
2. `browser_navigate` 到 `q.stock.sohu.com/cn/{代码}/index.shtml` — 最详细
3. `web_extract` 从东方财富/新浪财经提取

**⚠️ 浏览器超时处理**：
- 如果 `browser_navigate` 超时/连接失败，**不要反复重试同一URL**
- 立即降级到 `web_search` 搜索实时价格
- 搜索关键词：`{代码} ETF 最新价格 盘中`

**⚠️ 价格时效性（血泪教训）**：
- 搜索结果价格可能有5-30分钟延迟
- 必须标注"截至XX:XX"或"估算"
- **无法确认最新价格时，告知用户而非编造**
- 开盘后30分钟和尾盘前必须重新确认价格
- ⚠️ `web_extract` 从搜狐提取的行情可能是**前一个交易日的盘中数据**，不是最新收盘价！必须用页面标题中的价格（如"人工智能ETF易方达:2.13"）作为收盘价，不要用正文里看起来过时的数字
- ⚠️ 用户会当场纠正价格错误（"XXX收盘价是YYY"），这说明用户看到了实时数据而我用了过时数据 — 每次给出价格前先double-check是否与用户说的一致
- ⚠️ 昨收 vs 现价必须分清：集合竞价结束后"今开"可能≠昨收，不要用昨收当现价

**⚠️ 推荐：腾讯财经 API 直连（最稳定，首选方法）**：
当 web_search 和 browser 都不给力时，**直接用 terminal + curl 获取实时行情**。这是最可靠的方法——子代理和 execute_code 中的 urllib 有时被环境限制，terminal curl 不会。

```bash
# 实时行情（批量，逗号分隔）— 在 terminal 中直接执行
curl -s "https://qt.gtimg.cn/q=sh518880,sz159819,sz159681,sh513000" \
  -H "User-Agent: Mozilla/5.0" | iconv -f gbk -t utf-8
```

```python
# Python 解析（execute_code 中使用）
# 字段用 ~ 分隔，关键索引：
# [1]=名称 [2]=代码 [3]=现价 [4]=昨收 [5]=今开
# [6]=成交量(手) [32]=涨跌幅% [33]=最高 [34]=最低 [37]=成交额(万)
parts = line.split("~")
name, code, price = parts[1], parts[2], parts[3]
prev_close, open_p = parts[4], parts[5]
change_pct, high, low = parts[32], parts[33], parts[34]
amount_wan = parts[37]

# ⚠️ 价格字段校验（防止索引偏移导致错误分析）
try:
    cur = float(price)
    lo = float(low)
    hi = float(high)
    yclose = float(prev_close)
    assert cur >= lo, f"现价{cur} < 最低价{lo}，字段索引偏移！"
    assert cur <= hi, f"现价{cur} > 最高价{hi}，字段索引偏移！"
    assert abs(cur / yclose - 1) < 0.15, f"现价{cur}偏离昨收{yclose}超15%，数据异常！"
except (ValueError, AssertionError) as e:
    print(f"⚠️ 价格校验失败: {e}，降级使用备选数据源")
```
⚠️ **价格字段校验（2026-06-28 踩坑）**：
解析后必须校验 `float(parts[3]) >= float(parts[34])`（现价 ≥ 最低价），否则说明索引偏移。
如果校验失败，改用字段 [3] 作为现价，不要使用 [34]（最低价）或 [33]（最高价）替代现价。
```

**⚠️ 持仓盈亏分析模板（用户经常问"我的成本X，什么时候卖"）：**

```
1. 计算加权均价（分批买入时）
   均价 = (价1×量1 + 价2×量2 + ...) / 总量
2. 当前盈亏% = (现价 - 均价) / 均价 × 100
3. 卖出方案分三档：保守(+0.3~0.5%保本上方) / 趋势(+1~2%) / 目标(+3~5%)
4. 移动止损 = 现价-2% 或 MA5（取更近的）
5. 补仓条件：不追高，等回调到MA10/MA20或放量企稳信号
```

**⚠️ 用户偏好（重要）：**
- 用户说"我不会止损的" → 不要强推止损，改为"移动止盈"话术（"让利润奔跑，设好止盈保护已赚的"）
- 用户问"什么时候可以补仓" → 给明确条件（回踩XX均线、放量收阳、买盘恢复），不给模糊说法
- 用户经常盘中问"能卖吗" → 直接给概率+建议，不要绕弯子
- 用户有多次分批买入记录 → 先算加权均价再分析

## ⚠️ 数据获取优先级（必读）

**用户要求分析时，永远先拿实时行情，再做技术分析。不要用K线历史数据替代实时价格！**

优先级（从高到低）：

2. web_search 搜索实时价格 — 次选
3. execute_code 中 urllib 腾讯API — 偶尔被环境限制
4. 浏览器访问 — 最慢，超时风险

**⚠️ 血泪教训（2026-06-24）**：用户要求分析4只ETF的买卖点位，我先返回了K线历史数据（截至前一日收盘），用户立即纠正"获取实时场内价格后再分析"。正确做法：先 terminal curl 拿实时价格，再用K线算支撑压力位，最后结合两者输出分析。

## 批量分析模式

当用户给出大量代码（10-40只）时：
1. **一个curl请求**拿到所有实时行情（不要分批、不要用子代理）
2. **一个execute_code**循环拿到所有K线数据
3. 按涨跌幅排序输出全景表
4. 分类（T+0商品/T+0跨境/T+1）后逐类给买卖点位
5. 最后给操作优先级速查表

**用户给出大量代码时不要问"要分析哪些"——全部分析，一个不少。**

## 持仓组合监控模式

当用户提供**多只ETF + 持仓成本 + 份数**时，按以下流程：

1. **一个curl请求**拿到所有实时行情（批量，逗号分隔）
2. **解析字段**：现价[3]、涨跌%[32]、买一价[9]、卖一价[19]、买一量[8]、卖一量[18]、最高[33]、最低[34]
3. **计算浮盈**：`(现价-成本)/成本×100%`，`浮盈额=(现价-成本)×份数`
4. **按浮亏金额排序**：亏损最大的排最前（优先级最高）
5. **输出格式**：每只ETF输出6项（现价涨跌/买卖一/建议/精确点位/止损价/卖点）
6. **最后输出**：操作优先级排序表 + 风险提示（最大亏损源合计占比）

**⚠️ 核心策略**：不认亏卖，做T降本为主。即使浮亏-37%也不建议清仓，而是做T降本。

详见 `references/portfolio-monitoring-template.md`。
